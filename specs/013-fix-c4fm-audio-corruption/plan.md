# Implementation Plan: Fix C4FM Dispatch Tone Corruption & Silent Audio Gaps

**Branch**: `013-fix-c4fm-audio-corruption` | **Date**: 2026-02-17 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/013-fix-c4fm-audio-corruption/spec.md`

## Summary

The upstream audio squeak fix (commit `de0e722f`) added a sync detection guard `if(mMessageAssembler == null)` to `P25P1MessageFramer.syncDetected()`. This guard prevents false syncs from corrupting in-progress message assembly, but it also blocks **legitimate** sync detections at message boundaries. When a new P25 frame sync arrives, the previous message's assembler is typically complete but not yet dispatched (dispatch happens on the next `symbol()` call). The guard silently drops the sync, preventing NID collection, which prevents `nidDetected()` → `forceCompletion()`, causing the stale assembler to accumulate junk symbols from the next message. This produces garbled IMBE voice frames (tone corruption) and missed LDUs (silent gaps).

**Fix**: Modify `syncDetected()` to dispatch any complete-but-not-yet-dispatched message before checking the guard. This allows legitimate syncs at message boundaries while maintaining the audio squeak protection for incomplete messages.

## Root Cause Analysis

### The Bug: Sync Detection Race Condition

In P25P1, frame structure is: `[SYNC 24 dibits] [NID 33 dibits] [Message Body] [SYNC] [NID] ...`

**Normal P25 message transition (pre-merge):**
1. Last body symbol of Message A → passed to framer → assembler completes
2. Sync pattern for Message B detected by demodulator
3. `syncDetected()` → `mSyncDetected = true`, `mNIDPointer = 0` (unconditional)
4. NID symbols collected (33 dibits)
5. NID validates → `nidDetected()` → `forceCompletion()` dispatches Message A → new assembler for Message B

**Broken flow (post-merge with guard):**
1. Last body symbol of Message A → assembler completes (`isComplete() == true`)
2. Sync pattern for Message B detected
3. `syncDetected()` → guard checks `mMessageAssembler == null` → **FALSE** (assembler is complete but not yet dispatched — dispatch happens on next `symbol()` call)
4. **Sync is silently dropped**
5. Next `symbol()` call dispatches Message A, nulls assembler
6. But sync was already lost — NID collection never starts
7. Stale state persists, voice frames from Message B are garbled or missed

### Why It's All-or-Nothing

The tone corruption is all-or-nothing because:
- If the sync at the START of a voice transmission is caught → entire LDU sequence decodes correctly
- If the sync is missed → the framer never re-syncs for that message sequence → all voice frames are corrupted
- Subsequent sync detections may also fail if the assembler stays non-null

### The Fix

```java
public void syncDetected()
{
    // If a message is complete but not yet dispatched, dispatch it now
    // to allow the new sync detection. This resolves the race condition
    // where sync detection arrives before the next symbol() call dispatches.
    if(mMessageAssembler != null && mMessageAssembler.isComplete())
    {
        dispatchMessage();
    }

    // Only allow sync detection if we're not currently assembling a message.
    // This preserves the audio squeak fix (de0e722f) by blocking false syncs
    // that arrive mid-assembly.
    if(mMessageAssembler == null)
    {
        mSyncDetected = true;
        mNIDPointer = 0;
        mSyncDetectionCount++;
    }
}
```

**Why this works:**
- Complete-but-undispatched messages are flushed before the guard check → legitimate syncs pass
- Incomplete messages still block false syncs → audio squeak fix preserved
- Multi-block sequences (TSBK1→2→3) create reconfigured assemblers (non-null, not complete) → correctly blocked
- Minimal change: 4 lines added to existing method
